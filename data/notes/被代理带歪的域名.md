一、问题现象

这次的问题一开始表现得很像“网站挂了”，我访问：https://cygee.top

浏览器报错：

ERR_CONNECTION_CLOSED
似乎 cygee.top 关闭了连接

但是很快我又发现一个很奇怪的现象：

https://www.cygee.top 可以打开
https://cygee.top 打不开

也就是说，不是整个网站都打不开，而是根域名 cygee.top 出问题，www 子域名还能访问。

一开始我能想到的怀疑点有这些：

1. DNS 配错了
2. Vercel 域名没有绑定好
3. HTTPS 证书有问题
4. 网站部署挂了
5. 本地代理影响了访问

因为 ERR_CONNECTION_CLOSED 更像是连接层的问题，不像普通的页面 500 或 React 报错。


二、第一轮判断：先查 DNS

我先在 PowerShell 里运行：

nslookup cygee.top
nslookup www.cygee.top

一开始查出来的结果非常关键：

cygee.top      -> 198.18.1.48
www.cygee.top  -> 198.18.2.214

这个结果不正常。

因为我的网站是部署在 Vercel 上的，根域名正常应该解析到类似：

76.76.21.21

而 198.18.x.x 很像代理软件的 Fake-IP 地址。

所以这一步可以判断：

我的域名没有直接解析到 Vercel，而是先被 Clash / TUN / Fake-IP 接管了。

用形象一点的话说：

我本来要去“Vercel 大楼”，正常导航应该告诉我真实地址是 76.76.21.21。
结果代理软件给我塞了一个临时中转门牌号 198.18.x.x，并告诉我：“你先来我这里，我帮你转过去。”
但是它没有转好，所以我就被卡在门口了。


三、第二轮判断：关闭 TUN / 清 DNS 后，DNS 正常了

后来我关闭了 TUN 或相关代理影响，并刷新 DNS：

ipconfig /flushdns

然后再次运行：

nslookup cygee.top

这次结果变成了：

cygee.top -> 76.76.21.21

www 也正常指向了 Vercel 的 CNAME：

www.cygee.top -> cname.vercel-dns.com

这说明 DNS 已经恢复正常了。

这一步的结论是：

腾讯云 DNS 和 Vercel 域名绑定大概率没有问题。
之前的异常主要是代理 DNS / Fake-IP 搞出来的。

我了解到：DNS 查询结果是排查网站打不开时非常关键的一层证据。如果查出来是 198.18.x.x，就应该优先怀疑代理软件，而不是先怀疑网站代码。


四、第三轮判断：用 curl 验证网站本身是否正常

接着我用命令行验证网站本身能不能访问：

curl.exe -I https://cygee.top

最后返回结果里有这些关键信息：

HTTP/1.1 200 OK
Server: Vercel
X-Matched-Path: /
X-Vercel-Cache: HIT

这一步非常关键。
这一步说明了：

1. cygee.top 这个域名本身是通的
2. Vercel 正常返回了页面
3. 网站首页可以正常响应
4. 这不是代码崩了，也不是 Vercel 项目挂了

就好比：
我打电话给店里，店员接了，说“我们正常营业”。
这说明店没有关。
如果我还是进不去，那问题大概率不在店，而在我去店里的路线、导航或者路上的检查站。


五、第四轮判断：浏览器还是打不开，说明是本机浏览器 / 代理链路问题

当时的状态变成了：

curl.exe 访问 cygee.top：正常
浏览器访问 cygee.top：不正常
关代理访问 cygee.top：正常
开代理访问 cygee.top：不正常

这个组合基本就能锁定问题：

不是网站问题，不是 DNS 主配置问题，不是 Vercel 部署问题
而是代理链路问题

也就是说，Clash Verge 的系统代理把浏览器请求接走了，然后代理节点访问我的 Vercel 域名失败。

这里我学到一个判断方法：
如果 curl.exe 已经能返回 200，但浏览器还是打不开，就不要继续盯着 Vercel 和代码了。
应该转向浏览器缓存、系统代理、代理规则、DNS 接管这些本机环境问题。


六、尝试过的修复 1：Fake IP 过滤

我后来在 Clash Verge 的 DNS 覆写里加了：

cygee.top, www.cygee.top, *.cygee.top

这一步的作用是让这些域名不要再被解析成 Fake-IP。

加完之后，nslookup 已经能正常返回：

cygee.top -> 76.76.21.21

说明 Fake-IP 这一层确实被修好了。

但是浏览器依然打不开。

这说明问题已经不在 DNS 解析层，而是在流量转发层。

这里要分清两个概念：

Fake IP 过滤：
解决“域名解析成假 IP”的问题。

系统代理绕过 / DIRECT：
解决“这个请求到底走不走代理”的问题。

用形象一点的话说：

Fake IP 过滤只是把导航地址改对了。
但是如果我的车还是被代理软件强制拉进代理通道，而代理通道本身是堵的，那我还是到不了。
所以还得告诉系统：“去 cygee.top 这条路，不要进代理通道，直接走。”


七、最终修复：系统代理绕过

最后我在 Clash Verge 的系统代理设置里找到了：

代理绕过设置

原本里面有：

localhost;127.*;192.168.*;10.*;172.16.*;...;

这些规则的意思是：访问这些地址时，不走系统代理，直接访问。

最后我在末尾加上：

;cygee.top;*.cygee.top

最终类似这样：

localhost;127.*;192.168.*;10.*;172.16.*;...;cygee.top;*.cygee.top

保存之后，访问 cygee.top 就不会再交给 Clash 的系统代理，而是直接访问 Vercel。

这才是最终修复。

也就是说：

Fake IP 过滤解决了“地址查错”的问题。
系统代理绕过解决了“路线走错”的问题。


八、整件事的核心原因

这次问题的最终原因可以总结成一句话：

我的 cygee.top 网站本身没坏，DNS 主配置也没坏；真正的问题是 Clash Verge 的代理 / Fake-IP / 系统代理规则影响了浏览器访问，导致请求没有稳定直连 Vercel。

更具体一点：

一开始：
cygee.top 被代理 DNS 解析成 198.18.x.x，这是 Fake-IP 接管。

后来：
DNS 恢复为 76.76.21.21，但是浏览器请求仍然走 Clash 系统代理。

最后：
把 cygee.top 加入系统代理绕过，让它不走代理，问题解决。


九、用一个完整比喻理解这件事

我可以把访问网站理解成“去一个地方”。

正常情况：

我输入 cygee.top
导航查地址
地址是 Vercel：76.76.21.21
我直接开车过去
成功到达

出问题时：

我输入 cygee.top
代理软件跳出来说：“别查真实地址，我给你一个中转地址 198.18.x.x”
浏览器去了这个中转地址
代理软件本来应该帮我转去 Vercel
但它没有转好
于是浏览器显示连接关闭

加 Fake-IP 过滤后：

导航地址查对了：76.76.21.21
但我的车还是被系统代理拉进代理通道
代理通道还是堵
所以还是打不开

最终加系统代理绕过后：

系统看到 cygee.top 在绕过名单里
不再把请求交给代理通道
而是直接访问 Vercel
访问成功


十、这次学到的排错模型

以后网站打不开时，我不能一上来就改代码，而应该按层排查：

1. 先看浏览器报错是什么

如果是 ERR_CONNECTION_CLOSED，更像连接层问题，不一定是代码问题。

2. 再查 DNS 结果

运行：

nslookup cygee.top

如果出现 198.18.x.x，就要优先怀疑 Clash Fake-IP / TUN / DNS 接管。

3. 再验证网站服务器本身通不通

运行：

curl.exe -I https://cygee.top

如果返回 200 OK，并且 Server 是 Vercel，说明网站本身大概率没有挂。

4. 比较 curl 和浏览器结果

如果 curl 正常，但浏览器打不开，重点查浏览器缓存、系统代理、代理规则。

5. 关代理做对照实验

如果关代理能打开，开代理打不开，那就说明不是网站问题，而是代理链路问题。

6. 最后用 DIRECT 或系统代理绕过解决

对于自己的域名，最稳的方式是加入系统代理绕过：

cygee.top;*.cygee.top

这样代理继续开着，但自己的博客域名不受代理影响。


十一、以后遇到类似问题要记住的几句话

nslookup 出现 198.18.x.x：
优先怀疑 Clash Fake-IP / TUN / DNS 接管。

curl.exe 返回 200，但浏览器打不开：
网站本身大概率没坏，应该查浏览器、代理、缓存。

关代理能访问，开代理不能访问：
代理链路有问题，给域名加 DIRECT 或系统代理绕过。

Fake-IP 过滤管“查到什么地址”。
系统代理绕过管“走哪条路”。
这两个不是一回事。

DNS 正常不等于访问链路正常。
解析层和路由层要分开看。
